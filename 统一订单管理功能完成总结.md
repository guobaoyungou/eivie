## 🎯 核心亮点

### 1. 数据标准化
所有订单类型统一为相同的数据格式，前端无需关心订单来源：
```json
{
  "id": 1001,
  "order_type": "shop",
  "order_type_name": "商城",
  "ordernum": "202603270001",
  "title": "商品名称",
  "cover_image": "封面图URL",
  "total_price": "99.00",
  "status": 0,
  "status_text": "待付款",
  "item_count": 2,
  "create_time": "2026-03-27 10:30",
  "detail_url": "/pagesExt/order/detail?id=1001"
}
```

### 2. 智能路由分发
根据订单类型自动跳转到对应详情页，无需手动判断：
- 商城订单 → `/pagesExt/order/detail?id=`
- 拼团订单 → `/activity/collage/orderdetail?id=`
- 选片订单 → `/pagesExt/order/detail?id=`
- ...

### 3. 性能优化
- **类型筛选直通**：选择特定类型时，只查询对应单表
- **应用层聚合**：避免数据库层UNION，提高查询效率
- **模块检测**：未开通的模块不参与查询

### 4. 易于扩展
新增订单类型只需4步：
1. 配置订单类型
2. 配置详情页路径
3. 添加标准化方法
4. 配置前端筛选和颜色

```javascript
// 获取全部订单

    pagenum: 1

    console.log(res.datalist);
});

// 获取商城订单
app.post('ApiUnifiedOrder/orderlist', {
    st: 'all',
    order_type: 'shop',
    keyword: '',
    pagenum: 1
}, function(res) {
    console.log(res.datalist);
});

// 搜索订单
app.post('ApiUnifiedOrder/orderlist', {
    st: 'all',
    order_type: 'all',
    keyword: '202603270001',
    pagenum: 1
}, function(res) {
    console.log(res.datalist);
});

// 获取订单数量统计
app.post('ApiUnifiedOrder/ordercount', {}, function(res) {
```

## 🔍 常见问题解答

### Q1：原有的订单列表页面还能用吗？
**A：** 可以。原有的各模块订单列表页面（如 `/activity/collage/orderlist`）和接口都保持不变，不受影响。

### Q2：选片订单为什么特殊处理？
**A：** 选片订单的数据表结构与其他订单不同（使用 `uid` 而不是 `mid`，且需要从 `order_goods` 表关联商品信息），因此需要特殊的标准化逻辑。

### Q3：如何确定一个订单类型是否开通？
**A：** 后端通过 `isModuleEnabled()` 方法检测，使用 `getcustom()` 函数读取模块开关状态。

### Q4：为什么使用应用层分页而不是数据库分页？
**A：** 因为数据来自多张表，如果使用数据库 UNION 分页会导致性能问题。应用层分页虽然会多查询一些数据，但整体性能更优。

### Q5：如何新增一个订单类型？
**A：** 参考《统一订单管理快速上手指南.md》的"如何新增订单类型"章节。

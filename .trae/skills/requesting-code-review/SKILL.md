# 请求代码审查（Requesting Code Review）

## 何时使用

此技能应该在以下情况下自动激活：
- 完成一个开发任务
- 准备提交代码
- 实施bug修复后
- 完成功能开发后
- 用户请求代码审查

## 核心原则

1. **双阶段审查**：先检查规范符合性，再检查代码质量
2. **按严重程度分类**：问题分为严重、重要、建议
3. **可操作反馈**：提供具体的改进建议
4. **建设性批评**：专注于改进，而非批评
5. **持续改进**：将审查结果用于学习和改进

## 审查流程

### 阶段1：规范符合性检查（Spec Compliance）

**目标**：确保代码符合计划和设计文档

**检查项**：

1. **功能完整性**
   - [ ] 所有计划的功能都已实现
   - [ ] 功能行为符合设计文档
   - [ ] 边界条件已处理
   - [ ] 错误处理已实现

2. **API/接口符合性**
   - [ ] API端点符合设计
   - [ ] 参数和返回值类型正确
   - [ ] 错误码和消息一致
   - [ ] 文档已更新

3. **数据模型符合性**
   - [ ] 数据模型符合设计
   - [ ] 字段类型和约束正确
   - [ ] 关系和索引已定义
   - [ ] 迁移脚本已创建

4. **测试覆盖**
   - [ ] 所有功能都有测试
   - [ ] 测试覆盖率达标（>80%）
   - [ ] 边缘情况已测试
   - [ ] 测试可以独立运行

**审查模板**：

```markdown
## 规范符合性审查

### 功能完整性
- ✅ 所有计划的功能都已实现
- ✅ 功能行为符合设计文档
- ⚠️ 边界条件需要加强：[具体说明]
- ✅ 错误处理已实现

### API符合性
- ✅ API端点符合设计
- ✅ 参数和返回值类型正确
- ⚠️ 错误码不一致：[具体说明]
- ✅ 文档已更新

### 测试覆盖
- ✅ 所有功能都有测试
- ❌ 测试覆盖率不足（65%，需要>80%）
- ⚠️ 边缘情况测试缺失：[具体说明]
- ✅ 测试可以独立运行

### 规范符合性总结
- 严重问题：0
- 重要问题：1
- 建议项：2

**必须修复**：测试覆盖率不足
```

### 阶段2：代码质量检查（Code Quality）

**目标**：确保代码质量高、可维护

**检查项**：

1. **代码风格**
   - [ ] 遵循项目代码规范
   - [ ] 命名清晰一致
   - [ ] 格式化正确
   - [ ] 没有代码异味

2. **代码复杂度**
   - [ ] 函数长度合理（<50行）
   - [ ] 圈复杂度低（<10）
   - [ ] 嵌套层次浅（<4层）
   - [ ] 没有重复代码（DRY）

3. **错误处理**
   - [ ] 异常被正确捕获
   - [ ] 错误消息清晰
   - [ ] 资源正确释放
   - [ ] 日志记录适当

4. **安全性**
   - [ ] 输入已验证
   - [ ] SQL注入防护
   - [ ] XSS防护
   - [ ] 认证和授权正确

5. **性能**
   - [ ] 没有明显的性能问题
   - [ ] 数据库查询优化
   - [ ] 缓存使用合理
   - [ ] 没有内存泄漏

6. **文档**
   - [ ] 公共API有文档
   - [ ] 复杂逻辑有注释
   - [ ] README已更新
   - [ ] 变更日志已记录

**审查模板**：

```markdown
## 代码质量审查

### 代码风格
- ✅ 遵循项目代码规范
- ⚠️ 命名不一致：`getUserData` vs `get_user_data`
- ✅ 格式化正确
- ⚠️ 代码异味：长函数（120行）

### 代码复杂度
- ❌ 函数过长：`process_data()` 120行（建议<50行）
- ⚠️ 圈复杂度高：`calculate_total()` 15（建议<10）
- ✅ 嵌套层次合理
- ⚠️ 重复代码：`validate_user()` 在多处重复

### 错误处理
- ✅ 异常被正确捕获
- ⚠️ 错误消息不清晰：`raise Exception("Error")`
- ✅ 资源正确释放
- ⚠️ 缺少日志记录

### 安全性
- ✅ 输入已验证
- ✅ SQL注入防护
- ⚠️ XSS防护不足：用户输入未转义
- ✅ 认证和授权正确

### 性能
- ⚠️ N+1查询问题：`get_items()` 循环查询数据库
- ✅ 缓存使用合理
- ✅ 没有内存泄漏

### 文档
- ✅ 公共API有文档
- ⚠️ 复杂逻辑缺少注释
- ⚠️ README未更新
- ❌ 变更日志未记录

### 代码质量总结
- 严重问题：2
- 重要问题：4
- 建议项：5

**必须修复**：函数过长、N+1查询问题
```

## 问题分类

### 严重问题（Critical）

**定义**：阻止代码合并或部署的问题

**示例**：
- 安全漏洞（SQL注入、XSS等）
- 数据丢失风险
- 严重的性能问题
- 测试完全缺失
- 违反核心业务规则

**处理**：必须修复才能继续

**模板**：

```markdown
### 🔴 严重问题

**1. SQL注入漏洞**
- **位置**：`app/api/users.py:45`
- **问题**：直接拼接SQL查询，存在注入风险
- **代码**：
  ```python
  query = f"SELECT * FROM users WHERE id = {user_id}"
  ```
- **修复**：使用参数化查询
  ```python
  query = "SELECT * FROM users WHERE id = %s"
  cursor.execute(query, (user_id,))
  ```
- **优先级**：必须立即修复
```

### 重要问题（Important）

**定义**：应该修复但不阻止合并的问题

**示例**：
- 代码复杂度高
- 错误处理不完整
- 测试覆盖率不足
- 性能问题
- 代码重复

**处理**：建议修复，可以合并但需要跟踪

**模板**：

```markdown
### 🟡 重要问题

**1. 函数过长**
- **位置**：`app/services/payment.py:120-240`
- **问题**：`process_payment()` 函数120行，难以维护
- **建议**：拆分为多个小函数
  ```python
  def process_payment(payment_data):
      validated = validate_payment_data(payment_data)
      authorized = authorize_payment(validated)
      executed = execute_payment(authorized)
      return executed
  ```
- **优先级**：建议在下次迭代修复
```

### 建议项（Suggestion）

**定义**：改进建议，不影响功能

**示例**：
- 命名改进
- 代码风格
- 文档改进
- 性能优化
- 重构建议

**处理**：可选，用于持续改进

**模板**：

```markdown
### 🔵 建议项

**1. 命名改进**
- **位置**：`app/utils/helpers.py:15`
- **当前**：`def calc(x, y):`
- **建议**：`def calculate_sum(x, y):`
- **原因**：更清晰、更具描述性
- **优先级**：可选
```

## 最佳实践

1. **具体明确**：提供具体的文件位置和代码示例
2. **建设性**：专注于如何改进，而非批评
3. **可操作**：提供具体的修复建议
4. **及时**：尽快完成审查，不要拖延
5. **学习机会**：将审查作为学习机会
6. **一致性**：使用一致的审查标准
7. **自动化**：尽可能使用自动化工具

## 自动化审查工具

### Python项目

```bash
# 代码风格检查
ruff check app/

# 类型检查
mypy app/

# 安全检查
bandit -r app/

# 复杂度检查
radon cc app/ -a

# 重复代码检查
pylint --disable=all --enable=duplicate-code app/
```

### TypeScript项目

```bash
# 代码风格检查
biome check src/

# 类型检查
tsc --noEmit

# 安全检查
npm audit

# 代码质量
eslint src/
```

## 审查后行动

### 如果通过

1. 合并代码
2. 更新文档
3. 记录变更
4. 通知团队

### 如果有条件通过

1. 修复重要问题
2. 重新提交审查
3. 验证修复
4. 合并代码

### 如果不通过

1. 修复严重问题
2. 重新提交审查
3. 全面验证
4. 再次审查

# AI旅拍人像上传门店选择功能完整修复

## 概述

本次修复解决了AI旅拍人像管理模块中的两个关键问题：
1. **人像上传时无法选择门店** - 前端缺少门店选择器
2. **门店下拉列表只显示"无门店"** - 后端查询逻辑错误

## 问题1：人像上传门店选择功能

### 问题描述
在人像管理列表中，上传的人像门店字段（mdid）为空值（0），因为上传时没有提供门店选择功能。

### 解决方案
在上传弹窗中添加门店选择器，允许用户为批量上传的人像指定门店。

**修改文件**: `/www/wwwroot/eivie/app/view/ai_travel_photo/portrait_list.html`

**关键改动**:
```javascript
// 1. 在弹窗中添加门店选择器
html += '<div style="margin-bottom: 15px;">';
html += '<label>选择门店：</label>';
html += '<select id="upload_mdid">';
html += '<option value="0">无门店</option>';
{foreach $mendian_list as $md}
html += '<option value="{$md.id}">{$md.name}</option>';
{/foreach}
html += '</select>';
html += '</div>';

// 2. 上传时传递门店ID
var mdid = $('#upload_mdid').val() || 0;
formData.append('mdid', mdid);

// 3. 上传中禁用门店选择
$('#upload_mdid').prop('disabled', true);
```

## 问题2：门店列表查询逻辑错误

### 问题描述
admin账号下实际有"绿万鸿花卉园艺"等3个门店，但上传弹窗的门店下拉列表只显示"无门店"。

### 问题原因
数据库分析显示：
- admin账号（aid=1）有17个门店
- 前3个门店（绿万鸿花卉园艺等）的`bid=0`（历史数据）
- 其他门店的`bid`为具体商家ID（10、11、12等）

原代码问题：
```php
// 超级管理员bid=0，转换为targetBid=1
$targetBid = $this->bid;
if ($targetBid == 0) {
    $targetBid = Db::name('business')->where('aid', $this->aid)->value('id');
}

// 使用targetBid=1查询，但门店bid=0，导致查不到
$mendian_list = Db::name('mendian')
    ->where('aid', $this->aid)
    ->where('bid', $targetBid)  // ❌ 查不到bid=0的门店
    ->select();
```

### 解决方案
修改门店查询逻辑，超级管理员获取所有门店，普通商家只获取自己的门店。

**修改文件**: `/www/wwwroot/eivie/app/controller/AiTravelPhoto.php`

**修改方法**: `portrait_list()`

**关键改动**:
```php
// 获取门店列表（超级管理员获取所有门店，普通商家只获取自己的门店）
$mendianWhere = [['aid', '=', $this->aid]];
if ($this->bid == 0) {
    // 超级管理员：获取aid下所有门店（包括bid=0的历史数据）
    // 不添加bid条件
} else {
    // 普通商家：只获取自己的门店
    $mendianWhere[] = ['bid', '=', $this->bid];
}

$mendian_list = Db::name('mendian')
    ->where($mendianWhere)
    ->select();
```

## 功能特性

### 1. 门店选择器
- **位置**: 上传弹窗顶部显著位置
- **选项**: 包含"无门店"和所有可用门店
- **默认值**: "无门店"（mdid=0）
- **交互**: 上传过程中禁用，防止误操作

### 2. 权限控制
- **超级管理员**: 可以看到所有门店（包括bid=0的历史门店）
- **普通商家**: 只能看到自己的门店
- **数据隔离**: 人像数据仍按targetBid隔离

### 3. 兼容性
- **历史数据**: 完全兼容bid=0的旧门店数据
- **新数据**: 支持新的门店数据结构
- **向后兼容**: 不影响现有功能和数据

## 数据流程

### 上传流程
```
1. 用户点击"批量上传人像"
   ↓
2. 弹窗显示门店选择器（加载所有可用门店）
   ↓
3. 用户选择门店（或选择"无门店"）
   ↓
4. 用户选择图片文件
   ↓
5. 点击"开始上传"
   ↓
6. 逐个上传图片，每个请求包含：
   - file: 图片文件
   - mdid: 用户选择的门店ID
   ↓
7. 后端保存人像记录，包含门店信息
   ↓
8. 列表刷新，显示门店名称
```

### 门店列表查询
```
1. 页面加载时查询门店列表
   ↓
2. 判断用户类型：
   - bid == 0 → 超级管理员
   - bid != 0 → 普通商家
   ↓
3. 构建查询条件：
   - 超级管理员: WHERE aid = ?
   - 普通商家: WHERE aid = ? AND bid = ?
   ↓
4. 返回门店列表给前端
   ↓
5. 前端渲染门店选择器
```

## 注意事项

### 1. 数据库结构
- `ai_travel_photo_portrait.mdid`: 门店ID，默认0表示无门店
- `mendian.bid`: 商家ID，历史数据可能为0
- 超级管理员的bid始终为0

### 2. 权限设计
- 超级管理员可以查看和操作所有门店的数据
- 普通商家只能查看和操作自己门店的数据
- 门店列表查询不影响数据权限隔离

### 3. 历史数据处理
- 无需修改数据库中的历史数据
- 历史门店（bid=0）在超级管理员账号下正常显示
- 新门店按正常流程创建（包含正确的bid）

### 4. 性能考虑
- 门店列表在页面加载时一次性查询
- 批量上传时逐个发送请求，避免大文件超时
- 门店数量较多时（>100）建议添加分页或搜索
